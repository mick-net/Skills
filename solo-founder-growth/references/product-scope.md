# Product Scope

Use this when deciding what to build, what not to build, or how to simplify an MVP.

## The Solo-Founder Build Filter

Build only when the feature passes at least one:

- Activation: helps a new user reach value faster.
- Conversion: makes a buyer confident enough to pay.
- Retention: makes repeated use more likely.
- Expansion: supports a higher-priced plan or team use case.
- Distribution: creates a shareable artifact, searchable page, integration, or public proof.
- Support reduction: removes repeated manual support burden.

If a feature does not pass one, cut it.

## Early-Stage Hard Stop

For `still-building/prelaunch`, `launched/no revenue`, and `first paying customers / $1-$1k MRR`, do not recommend multi-day implementation unless these are defined:

- One buyer.
- One painful current workaround.
- One reachable channel or 10 prospects.
- One public promise or demo.
- One paid ask: checkout, invoice, pilot, paid setup, paid report, or manual service.

If any are missing, recommend validation, outreach, public-surface work, or manual fulfillment before code.

## Manual First

When the founder asks for automation, propose the manual or concierge version first unless repeated usage, support load, or paid demand already proves the workflow.

## MVP Shape

A useful MVP is a finished simpler product, not a half-built version of the full vision.

Prefer:
- One specific audience.
- One painful job.
- One obvious promise.
- One paid plan or one clear paid boundary.
- One repeatable acquisition path.
- Manual or semi-manual delivery before automation.

Avoid:
- Multi-person permissions before one user loves it.
- Dashboards before users know what action to take.
- Integrations before the core job is valuable.
- Mobile apps when a web app or desktop app is enough.
- AI agents/workflows before a simple tool solves the job.
- Complex onboarding that explains instead of removing friction.

## Weekend-Scope Test

A good first version can usually be one of:

- A form plus manual fulfillment.
- A searchable directory.
- A single-purpose utility.
- A template, boilerplate, or workflow pack.
- A static app with local storage.
- A BYO-key tool.
- A narrow integration.
- A paid report, audit, or concierge service.

## Product Wedge Formula

Use:

`For [specific audience], [product] helps [painful job] by [fast mechanism], without [hated alternative].`

Examples:
- For solo SaaS founders, an analytics tool shows which channels create paying customers, without GA complexity.
- For AI power users, a chat UI stores prompts and conversations locally, without another monthly AI subscription.

## Expansion Timing

Expand only after one of these is true:

- Users repeatedly ask for the same adjacent workflow.
- A paid tier cannot close without the feature.
- Support volume proves the manual process needs automation.
- The feature opens a concrete channel, integration marketplace, or buyer segment.
- A team/B2B buyer asks for security, admin, billing, or deployment features and is willing to pay.

## Scope Output Template

Use this structure when advising:

- `Build now`: 1-3 items that directly improve the bottleneck.
- `Do manually`: steps the founder can fulfill before automating.
- `Build later`: useful after traction proof.
- `Do not build`: attractive distractions.
- `Proof required`: the customer behavior that would justify expansion.
